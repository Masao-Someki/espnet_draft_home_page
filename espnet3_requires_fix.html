    ESPnet3 Documentation and Codebase Fix Requirements

    Date: 2026-05-20

    Scope Checked

    This review covered:

    • espnet3/: 53 Python files, including systems, components, publication, parallel execution, and utilities.
    • egs3/: TEMPLATE/asr, mini_an4/asr, librispeech_100/asr, and partial TTS dataset examples.
    • doc/vuepress/src/espnet3: 45 Markdown files, 6,923 total lines.
    • test/espnet3: 39 ESPnet3-specific test files.
    • Official documentation patterns from PyTorch and Hugging Face.

    External references checked:

    • PyTorch API docs: https://docs.pytorch.org/docs/2.12/index.html
    • PyTorch tutorials: https://docs.pytorch.org/tutorials/
    • Hugging Face Transformers docs: https://huggingface.co/docs/transformers/index
    • Hugging Face Hub docs: https://huggingface.co/docs/hub/index
    • Hugging Face model cards: https://huggingface.co/docs/hub/model-cards
    • Hugging Face Spaces: https://huggingface.co/docs/hub/spaces-overview

    Current ESPnet3 State

    ESPnet3 already has a meaningful implementation surface. It is not just a stub:

    • Core stage APIs exist under espnet3/systems/base.
    • ASR support exists under espnet3/systems/asr.
    • Dataset orchestration exists under espnet3/components/data.
    • Lightning integration exists under espnet3/components/modeling and espnet3/components/trainers.
    • Inference, metric, publication, demo, writer, config, logging, and parallel helpers exist.
    • There is a usable egs3/TEMPLATE/asr recipe structure.
    • mini_an4/asr and librispeech_100/asr contain concrete ASR examples.
    • ESPnet3 tests exist and cover many components.

    The main issue is not lack of material. The issue is that docs, code names, recipe flags, and generated artifacts are not aligned enough for users or external contributors.

    Critical Fixes

    1. espnet3.utils.publish is referenced but does not exist

    The code imports espnet3.utils.publish, but the actual file is espnet3/utils/publish_utils.py.

    Observed references:

    • espnet3/systems/base/system.py
    • espnet3/publication/demo/packing.py
    • espnet3/publication/inference_model.py docstrings
    • Multiple docs refer to publish helpers indirectly.

    Confirmed failure:

    python -c 'import espnet3.utils.publish'

    Result:

    ModuleNotFoundError: No module named 'espnet3.utils.publish'

    Required fix:

    • Choose one canonical module name.
    • Prefer espnet3.utils.publish_utils unless there is a strong reason to rename the file.
    • Update imports, docstrings, docs, and tests.
    • Do not add a compatibility wrapper unless explicitly required. This is local ESPnet3 refactor work.

    2. Stage names are inconsistent: metric vs measure

    The implementation uses measure.

    Examples:

    • egs3/TEMPLATE/asr/run.py has DEFAULT_STAGES with measure.
    • espnet3/systems/base/metric.py defines measure(metrics_config).
    • BaseSystem exposes measure().

    The docs often use metric.

    Examples:

    • doc/vuepress/src/espnet3/get_started.md
    • doc/vuepress/src/espnet3/evaluate.md
    • doc/vuepress/src/espnet3/stages/metrics.md
    • doc/vuepress/src/espnet3/config/index.md
    • doc/vuepress/src/espnet3/recipe_directory.md

    Required fix:

    • Decide whether the public stage is measure or metric.
    • If measure is the real stage, update all user-facing docs, examples, tables, and command lines.
    • If metric is desired, rename the implementation and recipe stages coherently.
    • Avoid maintaining both names unless explicitly requested.

    3. CLI config flag names are inconsistent

    The actual egs3/TEMPLATE/asr/run.py parser uses:

    • --training_config
    • --inference_config
    • --metrics_config
    • --publication_config
    • --demo_config

    Many docs use older or shorter names:

    • --train_config
    • --infer_config
    • --metric_config
    • --publish_config

    Affected docs include:

    • doc/vuepress/src/espnet3/get_started.md
    • doc/vuepress/src/espnet3/recipe_directory.md
    • doc/vuepress/src/espnet3/config/index.md
    • doc/vuepress/src/espnet3/stages/train.md
    • doc/vuepress/src/espnet3/stages/inference.md
    • doc/vuepress/src/espnet3/stages/metrics.md
    • doc/vuepress/src/espnet3/stages/publish.md
    • doc/vuepress/src/espnet3/stages/create-dataset.md
    • doc/vuepress/src/espnet3/stages/collect-stats.md
    • doc/vuepress/src/espnet3/stages/system-specific.md

    Required fix:

    • Pick the public CLI flag names and make every recipe and doc match.
    • If the existing parser names stay, docs must use --training_config, --inference_config, --metrics_config, and --publication_config.
    • Update error-message examples in docs too.

    4. Metric class names are stale in docs

    The implementation uses:

    • espnet3.components.metrics.base_metric.BaseMetric
    • espnet3.systems.base.metric.measure

    Some docs still refer to:

    • espnet3.components.metrics.abs_metric.AbsMetric
    • espnet3.systems.base.metric.metric

    Affected docs:

    • doc/vuepress/src/espnet3/evaluate.md
    • doc/vuepress/src/espnet3/stages/metrics.md

    Required fix:

    • Replace AbsMetric with BaseMetric.
    • Replace metric() with measure() if the stage remains measure.
    • Update code examples and imports.

    5. Inference function name is stale in docs

    The implementation uses:

    • espnet3.systems.base.inference.infer

    Some docs refer to:

    • espnet3.systems.base.inference.inference
    • def inference(config: DictConfig):

    Affected docs:

    • doc/vuepress/src/espnet3/evaluate.md
    • doc/vuepress/src/espnet3/stages/inference.md

    Required fix:

    • Replace inference() with infer() where referring to ESPnet3's base entrypoint.
    • Keep "inference" as the concept and config terminology.

    6. Demo docs describe a removed espnet3/demo package

    The actual implementation is under:

    • espnet3/publication/demo/assets.py
    • espnet3/publication/demo/packing.py
    • espnet3/publication/demo/session.py

    Docs still refer heavily to:

    • espnet3/demo/pack.py
    • espnet3/demo/setup.py
    • espnet3/demo/app_builder.py
    • espnet3/demo/ui.py
    • espnet3/demo/runtime.py
    • espnet3/demo/resolve.py

    Affected docs:

    • doc/vuepress/src/espnet3/core/demo.md
    • doc/vuepress/src/espnet3/core/index.md
    • doc/vuepress/src/espnet3/core/components/demo/app_builder.md
    • doc/vuepress/src/espnet3/core/components/demo/pack_pipeline.md
    • doc/vuepress/src/espnet3/core/components/demo/runtime.md
    • doc/vuepress/src/espnet3/core/components/demo/ui_definition.md
    • doc/vuepress/src/espnet3/core/systems.md

    Required fix:

    • Rewrite the demo docs around the current espnet3.publication.demo API.
    • Remove pages that describe deleted modules.
    • Document the current DemoSession, UIAsset, UIAssetRegistry, pack_demo, and upload_demo flow.
    • Align config examples with egs3/TEMPLATE/asr/conf/demo.yaml.

    7. decode terminology remains in new ESPnet3 docs

    The ESPnet3 naming rule says to avoid decode for inference output concepts.

    Remaining doc examples:

    • doc/vuepress/src/espnet3/config/publish_config.md
    • doc/vuepress/src/espnet3/stages/publish.md
    • doc/vuepress/src/espnet3/get_started.md uses "Runner-based decoding"
    • doc/vuepress/src/espnet3/core/demo.md uses "decoding args"

    Required fix:

    • Replace decode_dir with inference_dir or a precise publication-specific key.
    • Replace "decoding" with "inference" where describing ESPnet3 inference output.
    • Check implementation config keys too. If decode_dir still exists in code or template configs, migrate it.

    8. VuePress navigation does not expose ESPnet3 user docs

    doc/vuepress/sidebars.yml does not include doc/vuepress/src/espnet3 as a user documentation section.

    Current sidebar has:

    • Tutorials
    • Demos
    • Recipes
    • Python API
    • Shell API

    ESPnet3 appears only indirectly under generated Python API docs as guide/espnet3, not as the hand-written ESPnet3 guide.

    Required fix:

    • Add a first-class ESPnet3 section to doc/vuepress/sidebars.yml.
    • Include at least:
      • Getting started
      • Installation
      • Recipe directory
      • Config overview
      • Stages
      • Core concepts
      • Publication and demos
      • Troubleshooting

    9. Some internal links are broken

    Detected missing internal links:

    • doc/vuepress/src/espnet3/core/components/metrics.md -> ../../stages/measure.md
    • doc/vuepress/src/espnet3/core/components/metrics.md -> ../../config/measure_config.md
    • doc/vuepress/src/espnet3/core/components.md -> ../config/measure_config.md
    • doc/vuepress/src/espnet3/core/components.md -> ../stages/measure.md
    • doc/vuepress/src/espnet3/core/systems.md -> ../stages/measure.md

    Required fix:

    • Either rename pages to measure.* consistently, or update links to existing metrics.md and metric_config.md.
    • Add a docs link check to CI or pre-commit.

    High-Priority Documentation Fixes

    10. get_started.md should be rebuilt as the primary path

    The current getting-started doc mixes:

    • quick install
    • source install
    • system dependencies
    • import-based execution
    • cloned repo execution
    • stages
    • src/ implementation
    • Hydra config behavior
    • extra links

    It is useful, but not reliable because command flags and stage names are stale.

    Required fix:

    • Make it a short, tested quickstart.
    • Use one canonical recipe, likely egs3/mini_an4/asr.
    • Include commands that can be copied as-is.
    • Show expected outputs after each stage.
    • Link to deeper docs instead of explaining everything inline.
    • Add a "last verified with" note or CI command.

    11. Add a task-based documentation layer

    PyTorch has a clear split between tutorials, recipes, API reference, distributed, performance, and developer notes. Hugging Face docs split quickstart, task guides, how-to guides, conceptual guides, and API reference.

    ESPnet3 docs currently split mostly by implementation package and stage. That helps developers, but users usually start with a task.

    Missing task guides:

    • Train ASR from a local manifest.
    • Train ASR from Hugging Face Datasets.
    • Run inference on a trained checkpoint.
    • Score WER/CER.
    • Package a model.
    • Upload a model to Hugging Face Hub.
    • Create a Gradio Space/demo.
    • Add a new dataset recipe.
    • Add a new metric.
    • Add a new system-specific stage.

    Required fix:

    • Add a tasks/ or tutorials/ area under doc/vuepress/src/espnet3.
    • Keep package-level docs under core/.
    • Link task guides to API and config references.

    12. Add a config reference that is generated or validated

    Current config pages are hand-written and already drifted from code.

    Required fix:

    • Define canonical config schemas or schema-like examples for:
      • training
      • inference
      • metrics
      • publication
      • demo
      • parallel
      • dataset references
    • Add tests that load every documented minimal YAML example.
    • Consider generating parts of the config reference from template configs.

    13. Add end-to-end verified examples

    The docs should have at least one full path:

    1. Create dataset.
    2. Train tokenizer.
    3. Collect stats.
    4. Train.
    5. Infer.
    6. Measure.
    7. Pack model.
    8. Pack demo.

    Required fix:

    • Use mini_an4/asr as the smoke-test path.
    • Put exact commands in docs.
    • Add expected output files for each stage.
    • Add a CI job or test that validates the command snippets, even if the full training path is small.

    14. Add troubleshooting and FAQ pages

    Common failure modes are not collected in one place.

    Missing topics:

    • Missing optional dependencies.
    • Dask not installed.
    • Gradio not installed.
    • Hugging Face auth and HF_TOKEN.
    • trust_user_code behavior for packed models.
    • Missing inference_dir.
    • Duplicate test names.
    • Missing SCP fields.
    • Metric input alias mismatch.
    • CUDA tensor returned as inference artifact.
    • Stage config not provided.
    • Resolver path failures.
    • Windows/WSL path behavior.

    15. Add a migration guide from ESPnet2 recipes to ESPnet3 recipes

    There is a short core/espnet2_to_espnet3.md, but users need a practical migration guide.

    Required fix:

    • Map ESPnet2 concepts to ESPnet3 concepts:
      • shell stages -> System stage methods
      • run.sh -> run.py
      • asr.sh configs -> training.yaml, inference.yaml, metrics.yaml
      • Kaldi data dirs -> dataset references and DataOrganizer
      • decoding -> inference
      • scoring -> measure
      • model packing -> publication config
    • Include one before/after recipe example.

    egs3/ Fixes

    16. Remove generated artifacts from recipe directories unless intentionally committed

    egs3/mini_an4/asr currently contains generated outputs:

    • data/
    • downloads/
    • downloads.tar.gz
    • exp/
    • logs/
    • many __pycache__/ files
    • checkpoints
    • TensorBoard event files
    • generated model pack contents
    • generated inference outputs and metrics

    This makes the recipe look non-source-controlled and blurs what users should edit.

    Required fix:

    • Decide whether mini_an4 is a checked-in tiny fixture or a clean recipe.
    • If it is a fixture, document exactly which generated files are intentionally committed.
    • If it is a normal recipe, remove generated artifacts and update .gitignore.
    • Do not leave __pycache__ files under egs3.

    17. egs3 task coverage is uneven

    Current examples:

    • TEMPLATE/asr
    • mini_an4/asr
    • librispeech_100/asr
    • partial TTS dataset files under ljspeech/tts and mini_an4/tts

    Required fix:

    • Clarify which egs3 recipes are supported.
    • Either complete TTS recipe docs or mark TTS as experimental.
    • Add a short README per recipe explaining status, tested stages, and required resources.

    18. Recipe README files are too thin

    Existing recipe READMEs do not give enough status and reproducibility information.

    Required fix:

    • Add per-recipe README sections:
      • supported stages
      • data source
      • exact commands
      • expected runtime
      • expected outputs
      • known limitations
      • tested environment

    espnet3/ Code Documentation Fixes

    19. Public API docs are better than before, but generated docs need routing

    Many public classes and functions now have docstrings. However, the hand-written VuePress docs do not consistently link to generated API docs.

    Required fix:

    • Link conceptual pages to generated API pages under guide/espnet3.
    • Add API links for:
      • BaseSystem
      • DataOrganizer
      • DataLoaderBuilder
      • InferenceRunner
      • InferenceProvider
      • BaseMetric
      • InferenceModel
      • publication/demo helpers
      • config and writer utilities

    20. Package naming and docs naming should be normalized

    Examples of current drift:

    • publish_utils.py vs docs/imports saying publish.
    • publication/demo vs docs saying demo.
    • measure implementation vs docs saying metric.
    • BaseMetric implementation vs docs saying AbsMetric.

    Required fix:

    • Make a naming table and apply it across code, docs, recipes, and tests.
    • Prefer names already present in implementation unless the public API decision says otherwise.

    21. Add an architecture overview diagram

    PyTorch and Hugging Face docs both make navigation easy by separating concepts and references. ESPnet3 needs one clear architecture page.

    Required fix:

    • Add a single page that shows:
      • recipe run.py
      • configs
      • BaseSystem
      • stage entrypoints
      • components
      • datasets
      • parallel runners
      • publication/demo outputs
    • Keep the diagram simple and link each box to the relevant doc page.

    Gaps Compared With PyTorch Docs

    PyTorch docs provide:

    • A visible installation path.
    • A "Learn the Basics" path.
    • Tutorials by skill level and domain.
    • Recipes for focused tasks.
    • API reference separated from tutorials.
    • Distributed and performance sections.
    • Developer notes, FAQ, reproducibility, environment variables, and extension guides.

    ESPnet3 is missing or weak in these areas:

    • No first-class ESPnet3 sidebar entry.
    • No clean beginner path after installation.
    • No stable recipe cookbook.
    • No tested command snippets.
    • No performance guide for dataloading, batching, GPU use, or Dask.
    • No distributed training guide with concrete local and cluster examples.
    • No clear API reference routing from user docs.
    • No troubleshooting hub.
    • No reproducibility page covering seeds, logs, snapshots, configs, and artifacts.

    Recommended additions:

    • espnet3/tutorials/quickstart_asr.md
    • espnet3/tutorials/local_dataset_asr.md
    • espnet3/tutorials/huggingface_dataset_asr.md
    • espnet3/how_to/distributed_training.md
    • espnet3/how_to/debug_stage_failures.md
    • espnet3/how_to/package_and_upload.md
    • espnet3/reference/config/*.md
    • espnet3/reference/api_links.md
    • espnet3/troubleshooting.md
    • espnet3/reproducibility.md

    Gaps Compared With Hugging Face Docs

    Hugging Face docs provide:

    • Product-level split across Transformers, Hub, Datasets, Spaces, and Inference.
    • Quickstarts and task guides.
    • Strong Hub publication guidance.
    • Model cards with metadata, intended use, limitations, datasets, and evaluation results.
    • Spaces documentation for deployment, visibility, hardware, and SDK choices.
    • Clear API references and ecosystem integration pages.

    ESPnet3 is missing or weak in these areas:

    • Model publication docs do not yet match Hugging Face model card expectations.
    • Demo docs do not clearly map to Hugging Face Spaces behavior.
    • No user-facing guide for InferenceModel.from_packed and InferenceModel.from_pretrained.
    • No complete model card template guidance for ESPnet3.
    • No clear docs for trust_user_code, bundled recipe code, and security implications.
    • No documented Hub metadata recommendations such as library_name, datasets, language, license, tags, and evaluation results.
    • No Space hardware or dependency guidance for Gradio demos.

    Recommended additions:

    • espnet3/publication/model_cards.md
    • espnet3/publication/hugging_face_hub.md
    • espnet3/publication/spaces.md
    • espnet3/publication/inference_model.md
    • espnet3/security/trust_user_code.md

    Suggested Rewrite Order

    Phase 1: Make docs truthful

    Fix immediately:

    • publish_utils import/name mismatch.
    • measure vs metric.
    • CLI flag names.
    • BaseMetric vs AbsMetric.
    • infer vs inference.
    • publication/demo vs demo.
    • decode_dir terminology.
    • broken internal links.

    Phase 2: Make docs navigable

    Fix next:

    • Add ESPnet3 to doc/vuepress/sidebars.yml.
    • Create a short ESPnet3 landing page.
    • Reorganize docs into:
      • Quickstart
      • Tutorials
      • How-to guides
      • Concepts
      • Config reference
      • API reference links
      • Publication and demos
      • Troubleshooting

    Phase 3: Make docs executable

    Add:

    • Markdown command-snippet tests or scripted smoke tests.
    • YAML example load tests.
    • Internal link checks.
    • A small end-to-end mini_an4 docs verification path.

    Phase 4: Make docs competitive

    Add:

    • Task guides.
    • Performance and distributed guides.
    • Hugging Face Hub and Spaces guides.
    • Model card guidance.
    • Reproducibility and debugging guides.

    Bottom Line

    ESPnet3 has enough code and docs to become a serious documented subsystem, but the current state is internally inconsistent. The first pass should not add more prose. It should make the existing docs truthful against the current code, expose them in VuePress navigation, and remove generated recipe artifacts or mark them as intentional fixtures.

    After that, the docs should move toward the structure used by PyTorch and Hugging Face: quickstart, task guides, how-to guides, concepts, API reference, publication/deployment, troubleshooting, and reproducibility.
