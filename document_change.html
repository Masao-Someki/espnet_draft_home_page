    ESPnet3 Documentation Structure

    Background and Design Intent

    Why this structure exists

    The current ESPnet3 documentation has a fundamental problem: it is written as a library reference rather than a user goal guide. Pages explain what the library has, not what users can accomplish with it. As a result, most users cannot find the information they need without already knowing what to look for.

    This structure is designed around two principles:

    1. Every user can reach any piece of information they need within 3 clicks from the Landing page (/espnet3/). "3 clicks" means: Landing → section index (1 click) → individual page (2 clicks) → information on that page (3 clicks if a sub-page under guides/).
    2. Each page is readable in 3–4 minutes, covering 3–5 focused topics. Pages that try to cover everything end up being read by nobody.

    What was missing

    An audit of the current site against 7 user personas and 96 information needs found that 76 out of 96 items had no corresponding page, and 13 more existed but were hard to reach. Only 7 items were both present and accessible within 3 clicks. The most underserved personas were finetune users (0/15 needs met) and ESPnet3 contributors (3/19 needs met).

    How content is divided

    LayerPurposeExamples
    get-started/Get something running as fast as possibleInstall, run a recipe
    concepts/Understand the mental model before writing codeWhy ESPnet3, what is a System
    guides/Accomplish a specific goal, persona-by-personaFinetune a model, migrate from ESPnet2
    stages/Reference for each pipeline stagetrain, infer, measure
    core/Deep reference for library internalsConfig keys, Provider/Runner
    contributing/Add new Systems, Stages, ComponentsDev setup, PR flow
    troubleshooting.mdFix things when they breakOOM, CUDA errors

    Navigation design

    The ESPnet3 section (/espnet3/ and below) uses a dedicated sidebar that replaces the global sidebar. This is necessary because the global sidebar is ESPnet2-centric and actively misleads ESPnet3 users. VuePress Theme Hope supports path-based sidebar switching.

    Search (Algolia DocSearch, planned migration from slimsearch) handles access to docstrings. Class names, method names, and parameter names are indexed at the page level so users can reach any docstring in one search query. This means the 3-click rule applies to navigated content; docstrings are treated as search-first content.

    concepts/system-stage-component.md — System, Stage, and Component

    ItemDetail
    Path/espnet3/concepts/system-stage-component
    Clicks from Landing2
    StatusNew
    Primary personasAll

    Intent: The single most important concept page. Before any user can make sense of training.yaml, run.py, or the API docs, they need to understand what System, Stage, and Component are and how they relate. Currently this is never explained anywhere.

    Content to include:

    • Architecture diagram: System orchestrates Stages; Stages use Components; Provider/Runner handles execution
    • System: the task entry point (ASRSystem, TTSSystem, ...), owns the stage list
    • Stage: one step in the pipeline (create_dataset, train, infer, ...), reads from a config section
    • Component: reusable building block (DataOrganizer, ESPnetLightningModule, ...), used inside Stages
    • How run.py wires them together
    • Mapping to PyTorch concepts for PyTorch users
    • Mapping to ESPnet2 concepts for ESPnet2 migrators

    What users get: The mental model that makes every other page in the docs make sense.

    Currently missing: Entire page does not exist.
