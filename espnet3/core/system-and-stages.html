    System and Stages

    About 3 min

    System and Stages

    Before you read ESPnet3 stage code, it helps to learn these two concepts first. They explain how run.py, training.yaml, and the Python implementation fit together.

    Note

    This page is the reference for how run.py, System classes, stages, and stage logs fit together.

    In one sentence

    ConceptRoleExamples
    SystemSystem entry point that owns stage methodsASRSystem, TTSSystem
    StageOne named pipeline step, implemented as a System methodcreate_dataset, train, infer

    System

    A System is the top-level System object. It receives loaded configs and exposes the stage methods that run.py can call.

    BaseSystem
  ├── ASRSystem
  ├── TTSSystem
  └── Recipe-local System override

    Typical built-in stages from BaseSystem and related base modules:

    StagePurposeDefault target
    create_datasetBuild or prepare dataset assetsdataset module DatasetBuilder
    collect_statsCollect normalization or feature statisticsespnet3.systems.base.training.collect_stats
    trainTrain the modelespnet3.systems.base.training.train
    inferRun inference and write outputsespnet3.systems.base.inference.infer
    measureCompute metricsespnet3.systems.base.metric.measure
    pack_modelPackage the trained modelespnet3.utils.publish_utils.pack_model
    upload_modelUpload packaged outputsespnet3.utils.publish_utils.upload_model
    pack_demo / upload_demoPackage and publish demo assetsespnet3.publication.demo.packing

    Each System receives multiple configs, and each stage uses the config it needs:

    ConfigCommon stages
    training_configcreate_dataset, collect_stats, train, pack_model
    inference_configinfer
    metrics_configmeasure
    publication_configpack_model, upload_model
    demo_configpack_demo, upload_demo

    The base constructor surface is:

    class BaseSystem:
    def __init__(
        self,
        training_config=None,
        inference_config=None,
        metrics_config=None,
        publication_config=None,
        stage_log_mapping=None,
        demo_config=None,
    ):
        ...

    Stages

    A Stage is just a named method on the System object. The names passed through --stages map directly to method names.

    python run.py \
  --stages create_dataset train infer \
  --training_config conf/training.yaml \
  --inference_config conf/inference.yaml

    Conceptually, stage execution looks like this:

    for stage in stages_to_run:
    fn = getattr(system, stage)
    fn()

    Stages are independent. If dataset preparation is already done, you can run only train. If you only want model outputs, you can run only infer.

    If you define a custom System, adding a method is enough to add a new stage. You do not need a special registration step.

    How run.py connects them

    run.py is the thin orchestration layer. It does not implement the actual training or inference logic.

    python run.py --stages train infer --training_config conf/training.yaml
      │
      ▼
load configs
      │
      ▼
instantiate System
      │
      ├─► train()
      └─► infer()

    The heavy logic lives inside the System and the objects it constructs.

    How this maps to the directory layout

    espnet3/
  systems/
    base/
      system.py        # BaseSystem: stage methods + common wiring
      training.py      # collect_stats(cfg), train(cfg)
      inference.py     # infer(cfg): writes SCP/artifact outputs
      metric.py        # measure(cfg): writes metrics.json
    asr/
      system.py        # ASRSystem overrides and ASR-specific stages
      tokenizers/      # ASR tokenizer preparation helpers
      metrics/         # ASR metrics
  components/          # reusable data/modeling/training pieces
  parallel/            # provider/runner runtime

egs3/<recipe>/<task>/
├── src/
│   └── system.py
├── run.py
└── conf/
    • espnet3/systems/ holds shared framework code.
    • egs3/<recipe>/<task>/run.py selects stages and instantiates the System.
    • egs3/<recipe>/<task>/src/system.py is the right place for recipe-local behavior when you need it.

    Keep shared logic in espnet3/systems/. Keep recipe-specific logic inside the recipe.

    If you add a new shared System, put it under espnet3/systems/<task>/ and add a matching recipe template under egs3/TEMPLATE/<task>/.

    For a recipe-only override, keep the class under egs3/<recipe>/<task>/src/. That keeps one-off behavior out of the shared system layer.

    Stage log mapping

    Each stage writes logs to a stage-specific output directory. By default, the mapping follows the config paths:

    StageDefault path source
    create_datasettraining_config.data_dir
    collect_statstraining_config.stats_dir
    traintraining_config.exp_dir
    inferinference_config.inference_dir
    measuremetrics_config.inference_dir
    pack_modeltraining_config.exp_dir
    upload_modeltraining_config.exp_dir
    pack_demodemo_config.pack.out_dir
    upload_demodemo_config.pack.out_dir

    If no stage-specific path is available, ESPnet3 falls back to training_config.exp_dir, then to <cwd>/logs.

    Set recipe- or system-specific mappings in the System constructor and pass them to super().__init__.

    from omegaconf import DictConfig

from espnet3.systems.base.system import BaseSystem


class MySystem(BaseSystem):
    def __init__(
        self,
        training_config: DictConfig | None = None,
        inference_config: DictConfig | None = None,
        metrics_config: DictConfig | None = None,
        publication_config: DictConfig | None = None,
        stage_log_mapping: dict | None = None,
        demo_config: DictConfig | None = None,
    ) -> None:
        super().__init__(
            training_config=training_config,
            inference_config=inference_config,
            metrics_config=metrics_config,
            publication_config=publication_config,
            stage_log_mapping={
                "train_tokenizer": "training_config.tokenizer.save_path",
                "infer_dev": "training_config.exp_dir",
                **(stage_log_mapping or {}),
            },
            demo_config=demo_config,
        )

    The keys are stage method names. The values are dotted paths resolved from the system instance. Use **(stage_log_mapping or {}) last when callers should be able to override the system defaults.

    Adding or overriding stages

    Stage names are exposed from run.py, but the behavior lives on the System.

    For a recipe-local custom stage:

    from espnet3.systems.asr.system import ASRSystem


class RecipeSystem(ASRSystem):
    def export_debug(self):
        output_dir = self.exp_dir / "debug_export"
        output_dir.mkdir(parents=True, exist_ok=True)
        return output_dir

    Then run.py can include export_debug in the recipe stage list and instantiate RecipeSystem.

    Use this boundary:

    • recipe-specific behavior: egs3/<recipe>/<task>/src/system.py
    • reusable task behavior: espnet3/systems/<task>/system.py
    • detailed stage wiring checklist: Adding a Stage
